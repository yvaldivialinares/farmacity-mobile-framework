> **Screen complexity note:** This is the main authenticated dashboard. It is a vertically scrollable screen (`home-scroll-view`) containing multiple sections: a personalized welcome header, a profile completion banner, a featured categories horizontal carousel, a prescription medications CTA, a banking promotions carousel, and a nearest pharmacy location card. Content in carousels and promotions is **dynamic** (server-driven) — locators for individual cards are documented as patterns/templates, not fixed selectors.

## Feature: Welcome Header

### Description
Displays a personalized greeting using the logged-in user's first name (e.g. "¡Hola, Vic!") and a profile completion banner encouraging the user to complete their profile for a better experience. The banner includes a "Completar perfil" CTA button.

### Usage
The greeting text is dynamic — it changes according to the authenticated user's first name. The profile completion banner is only visible when the user's profile is incomplete. Tapping "Completar perfil" navigates to a profile completion screen. Neither the greeting nor the banner title has a `resource-id` or `content-desc` — use text-based XPath or parent-based structural locators for assertions.

### Elements & Locators

| Element | Type | Priority 1 | Priority 2 |
|---------|------|------------|------------|
| Greeting text | Text | `xpath: //android.widget.TextView[@text="¡Hola, {user_first_name}!"]` # FRAGILE: dynamic name in text — use partial match (`contains`) in implementation | — |
| Profile completion banner title | Text | `xpath: //android.widget.TextView[@text="Completá tu perfil"]` # FRAGILE: text-based | — |
| Complete profile button | Button/ViewGroup | `accessibility id: Completar perfil` | `xpath: //android.view.ViewGroup[@content-desc="Completar perfil"]` |

> **Greeting text** is the primary post-login validation element — confirms user is authenticated and data loaded. Use `contains(text(), "¡Hola,")` for stable assertions in automation.
> **Profile completion banner** may not appear if the user's profile is already complete. Do not assert its presence in smoke tests unless the test account is known to have an incomplete profile.

### Expected States/Validations
- Success: Greeting text visible containing "¡Hola," + user first name → confirms successful login
- Success: "Completar perfil" button visible when user profile is incomplete
- Failure: Greeting absent → login failed or wrong screen

## Feature: Bottom Navigation Bar

### Description
A persistent bottom navigation bar always visible across the authenticated area of the app. Contains 5 navigation items: Inicio, Pedidos, Comprar medicamentos, Mi perfil, Menú. The "Comprar medicamentos" button has a `resource-id` (`medicamentos-button`) — the others rely on `content-desc` only.

Each button navigates to its corresponding section. "Inicio" is the currently active tab when on home-screen. Tapping a non-active tab navigates to the corresponding screen. Bottom nav is persistent — it does not change between screens in the authenticated flow.

### ⚠️ Known Limitations
- **Greeting text is dynamic** — "¡Hola, Vic!" contains user-specific first name. Use `contains(text(), "¡Hola,")` assertion or `{user_first_name}` parameter; do not assert exact string unless test data is fixed.
- **All 10 `leads_to` destinations are unmapped** — this screen opens to the full authenticated area of the app. Priority mapping order recommended: `profile-screen` → `orders-screen` → `categories-screen` → `search-results-screen`.
- **Category cards and promotion cards** have no stable `resource-id` — rely solely on `content-desc` which is dynamic (server-driven). Test content-sensitive assertions with caution in CI.
- **Profile completion banner** only appears when user profile is incomplete — do not assert in smoke tests unless using a known incomplete-profile test account.
- **Search bar** has no `resource-id` or `content-desc` — text placeholder xpath only. Investigate if the search bar has a parent with a stable resource-id to use as a more robust structural locator.
- **Bottom nav `BottomNavPage`** component is recommended as a shared base — all authenticated screens will have the same 5 tabs with the same locators.
